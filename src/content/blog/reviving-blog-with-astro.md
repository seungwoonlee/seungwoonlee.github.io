오랫동안 방치되어 있던 `seungwoonlee.github.io` 리포지토리를 다시 꺼냈습니다. 단순히 글을 쓰는 공간을 넘어, **바이브 코딩(Vibe Coding)**의 정수를 기록할 현대적인 블로그로 탈바꿈시키기 위한 여정을 요약합니다.

## 1. 기술 스택: 왜 Astro인가?

기존의 정적 HTML 방식에서 벗어나 **Astro**를 선택한 이유는 명확합니다.

- **Content-first**: 마크다운 기반의 포스트 관리가 매우 직관적입니다.
- **Speed**: 필요한 만큼의 자바스크립트만 전송하여 압도적인 성능을 보여줍니다.
- **Developer Experience**: Tailwind CSS와의 매끄러운 연동 덕분에 스타일링 고민 없이 '바이브'에 집중할 수 있습니다.

## 2. 구축 과정의 기록

### 프로젝트 초기화
Astro의 공식 블로그 템플릿을 기반으로 구조를 잡았습니다. 불필요한 샘플 포스트들을 과감히 정리하고, 나만의 정체성을 담은 `src/consts.ts` 설정을 마쳤습니다.

### 스타일링: Tailwind CSS
`astro add tailwind` 명령 한 번으로 최신 Tailwind 환경을 구축했습니다. 이제 유틸리티 클래스만으로 세련된 UI를 빠르게 구현할 수 있습니다.

## 3. 트러블슈팅: 이미지 경로 문제 (ImageNotFound)

빌드 과정에서 예상치 못한 복병을 만났습니다. 

```bash
[ImageNotFound] Could not find requested image `/blog-placeholder-about.jpg`.
```

Astro의 최신 버전에서는 콘텐츠 파일 내에서 이미지를 참조할 때 **상대 경로**를 명확히 지정해줘야 한다는 점을 배웠습니다. 절대 경로(`/`) 대신 `../../assets/`와 같은 상대 경로로 수정하여 빌드 오류를 말끔히 해결했습니다. 역시 실전에서는 에러 메시지를 꼼꼼히 읽는 것이 중요합니다.

## 4. 자동화: GitHub Actions 도입

매번 수동으로 빌드하고 업로드하는 번거로움을 없애기 위해 **GitHub Actions** 워크플로우를 설정했습니다. 이제 `master` 브랜치에 코드를 푸시하기만 하면 깃허브 서버가 알아서 빌드하고 배포합니다.

## 마치며

오늘의 리뉴얼을 통해 '바이브 코딩'을 위한 든든한 베이스캠프가 마련되었습니다. 기술적인 노하우뿐만 아니라, 개발자로서의 고뇌와 성장을 이 공간에 꾸준히 담아보려 합니다.
